 (require serval/lib/core) package: serval

Serval provides a library for writing verifiers and specifications. This includes support for bug reporting, overflow checking for bitvectors, a memory model for low-level systems, symbolic optimizations, and common specifications.

2.1 Bug reporting

procedure

(bug-on cond [#:key key #:dbg dbg #:msg msg]) → void?

  cond : boolean?
  key : any/c = #f
  dbg : any/c = #f
  msg : string? = "Unknown bug-on"
Inserts a check that cond must be false under the current path condition, using key and dbg to provide optional information on bug type and location. This is useful for a verifier to add checks for undefined behavior,

procedure

(bug [#:key key #:dbg dbg #:msg msg]) → void?

  key : any/c = #f
  dbg : any/c = #f
  msg : string? = "Unknown bug"
Equivalent to (bug-on #t).

Clears checks produced by bug-on.

Formats a bug report for data produced by bug-on using a satisfiable solution.

2.3 Memory model

Serval provides a memory model suitable for low-level systems. It represents memory as a set of disjoint blocks. Each block may be an array, a struct, or a cell. Recursively, an array is a list of blocks of the same types; a struct is a list of named fields, where each field may further refer to a block. A cell is a leaf node: it is represented using an uninterpreted function.

To read and write with an offset into a block, one first invokes mblock-path to convert an offset to a list of indices, or a path; mblock-iload and mblock-istore! use a path to access the leaf cell.

2.4 Symbolic optimizations

The memory model internally invokes symbolic optimizations to simplify offsets and blocks. The following are two symbolic optimizations for case splitting.

procedure

(split-cases expr cases proc) → any?

  expr : any/c
  cases : list?
  proc : procedure?
Splits an expression for every concrete value given by cases, applies every value to proc, and returns the merged results.

syntax

(split-pc [struct field] expr ...)

Splits the expression for every possible concrete ite value and updates the struct field for subsequent evaluation.

2.5 Common specifications

Serval provides reusable definitions of common specifications, including refinement, noninterference, and reference counting.

2.5.1 Refinement

procedure

(verify-refinement #:implstate impl-state    
  #:impl impl-func    
  #:specstate spec-state    
  #:spec spec-func    
  #:abs abs-function    
  #:ri rep-invariant    
  [args    
  ce-handler]) → void?
  impl-state : any/c
  impl-func : procedure?
  spec-state : any/c
  spec-func : procedure?
  abs-function : procedure?
  rep-invariant : procedure?
  args : list? = null
  ce-handler : procedure? = (lambda (sol) (void))
Verifies state-machine refinement for a given implementation state, implementation function, specification state, specification function, abstraction function, and representation invariant, with args passed to both specification and implementation functions. One may optionally provide a counterexample handler when verification fails.

This helper function combines three checks: that the implementation doesn’t trigger undefined behavior, that the representation invariant holds on the resulting state of running the implementation if it holds before, and that the specification and implementation move in lock-step.

2.5.2 Noninterference

Serval provides the following definitions for unwinding conditions, a standard approach for proving noninterference.

procedure

(check-step-consistency #:state-init init-state    
  #:state-copy state-copy    
  #:unwinding unwinding    
  spec    
  [args]) → void?
  init-state : procedure?
  state-copy : procedure?
  unwinding : procedure?
  spec : procedure?
  args : list? = null
Verifies the step consistency property. Intuitively, giving any two states s1 and s2 that are equivalent with respect to an unwinding relation, the resulting states after executing spec remain equivalent: (unwinding s1 s2) implies (unwinding (spec s1) (spec s2)).

  #:invariants inv    
  #:dom dom    
  #:u u    
  #:unwinding unwinding    
  #:flowsto flowsto    
  action    
  spec    
  [args]) → void?
  init-state : procedure?
  state-copy : procedure?
  inv : procedure?
  dom : procedure?
  u : any/c
  unwinding : procedure?
  flowsto : procedure?
  action : any/c
  spec : procedure?
  args : list? = null
Verifies the weak step consistency property, as described in Nickel [3]. This is a more general version of step consistency for an intransitive information-flow policy. It is sometimes referred to as “confidentiality.”

procedure

(check-local-respect #:state-init init-state    
  #:state-copy state-copy    
  #:invariants inv    
  #:dom dom    
  #:u u    
  #:unwinding unwinding    
  #:flowsto flowsto    
  action    
  spec    
  [args]) → void?
  init-state : procedure?
  state-copy : procedure?
  inv : procedure?
  dom : procedure?
  u : any/c
  unwinding : procedure?
  flowsto : procedure?
  action : any/c
  spec : procedure?
  args : list? = null
Verifies the local respect property, as described in Nickel [3]. Intuitively, it checks that the states before and after an action are equivalent to a user u with respect to a policy. It is sometimes referred to as “integrity.”

2.5.3 Reference counting

The following functions support a specification of reference-counting consistency, which states that the reference count of a shared object (e.g., a file) equals the size of the set of owners (e.g., per-process file descriptors that refers to the file).

procedure

(init-refcnt) → refcnt?

Returns an new reference counter with a zero reference count.

procedure

(make-havoc-refcnt) → refcnt?

Returns an new reference counter with a symbolic reference count.

procedure

(incr-refcnt refcnt owner object) → refcnt?

  refcnt : refcnt?
  owner : any/c
  object : any/c
(decr-refcnt refcnt owner object) → refcnt?
  refcnt : refcnt?
  owner : any/c
  object : any/c
Return a new reference counter by increasing or decreasing the reference count by one.

procedure

(refcnt-invariants refcnt    
  owner-valid?    
  object-valid?    
  max-refs    
  owned-by?) → boolean?

Returns a predicate of reference-counting consistency, along with necessary invariants. It uses the encoding as described in Hyperkernel [2].
